# Customizing Enums: _missing_, _generate_next_value_, and member methods

> **Phase:** OOP Advanced & Modeling  •  **Stage:** 16.8 of 8  •  **Type:** `edu`  •  **Status:** skeleton (to be populated)

## What you'll learn
- Use _generate_next_value_ to make auto() produce meaningful, stable string codes (e.g. 'draft', 'partially_paid') instead of relying on positional integers, and understand that it must be defined BEFORE the members and is called once per member.
- Implement _missing_ as a classmethod so EnumType(value) accepts messy external input — case-insensitive names, surrounding whitespace, and known aliases — returning a canonical member or None to fall through to ValueError.
- Attach real domain behavior to enum members: a label @property for display, and methods like is_terminal() and can_transition_to(other) that encode the sales/payment state machine.
- Distinguish enum MEMBERS from ordinary class attributes (a transition table) so helper data living on the class does not accidentally become a member.
- Know when EnumType(x) raises ValueError vs. when _missing_ rescues it, and write code that relies on that contract.

## Python features introduced
`enum._generate_next_value_ (staticmethod controlling auto())`, `enum.auto`, `enum.StrEnum`, `enum._missing_ (classmethod fallback for unknown values)`, `adding ordinary methods to enum members (def on the class body)`, `adding @property to enum members`, `class-level (non-member) constants on an Enum via the typing/_ignore_ or plain dict pattern`, `member identity comparison with 'is' / 'in' over a tuple of members`, `iterating an enum (for m in cls) and reading .name / .value / .label`, `raising/catching ValueError from EnumType.__call__ when _missing_ returns None`, `str | None (PEP 604) return annotations`, `frozenset / dict transition tables keyed by enum members`

## MiniERP increment
Replaces the loosely-typed status strings used by Sales/Invoicing and Payments with two real domain enums in the shared core. OrderStatus and PaymentStatus are StrEnums whose wire codes are generated by _generate_next_value_ (so auto() yields 'draft', 'confirmed', 'shipped', 'cancelled' / 'unpaid', 'partially_paid', 'paid', 'refunded' — the exact strings persisted to JSON/CSV). _missing_ lets the Import/Export module ingest sloppy third-party status columns ('  Shipped ', 'CONFIRMED', legacy alias 'void' -> CANCELLED) case-insensitively and normalise them to canonical members, raising ValueError on genuinely unknown statuses so bad rows are rejected loudly. Member methods give the service layer its state machine: status.label for human-facing display/reports, status.is_terminal() to block edits to finished orders, and status.can_transition_to(target) so Sales/Payments service functions validate every status change against a single source of truth instead of scattered if/elif chains. This is the canonical status vocabulary that later phases (Reporting filters, Web API serialisation, Audit log entries) all reuse.
